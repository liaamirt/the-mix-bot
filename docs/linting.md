Документація процесів контролю якості коду (Linting & Typing)
Цей документ описує стандарти статичного аналізу, типізації та автоматизації, впроваджені в проєкті Telegram-бота the-mix-bot.

1. Інструменти аналізу коду
У проєкті використовується Ruff як універсальний інструмент для лінтингу та форматування коду. Він замінює собою flake8, isort та black.

Запуск перевірки: ruff check.

Запуск автовиправлення: ruff check --fix.

Форматування: ruff format.

2. Статична типізація (Static Type Checking)
Проєкт використовує Mypy у суворому режимі (strict = true) для забезпечення передбачуваності даних та мінімізації помилок у рантаймі.
Усі нові функції повинні мати анотації аргументів та типу повернення. Для роботи зі складними словниками (наприклад, даними бронювання з FSM), використовуються структури TypedDict з бібліотеки typing. Об'єкти даних Telegram Bot API (такі як Message, FSMContext) інтегровані завдяки вбудованим stubs бібліотеки aiogram.

Запуск перевірки типів: mypy.

3. Git Hooks (Pre-commit)
Для запобігання потраплянню неякісного коду до репозиторію налаштовано локальні хуки за допомогою фреймворку pre-commit.

Конфігурація знаходиться у файлі .pre-commit-config.yaml.

Усі коміти автоматично проходять перевірку Ruff (аналіз та форматування) та Mypy.

Якщо код містить порушення, створення коміту блокується до усунення проблем.

**Встановлення хуків для нових розробників (одноразова дія після клонування репозиторію):**
pre-commit install

4. Інтеграція з процесом збірки (CI/CD)
Безперервна інтеграція забезпечується через GitHub Actions. Робочий процес (workflow) ci.yml запускається на кожну подію push та pull_request у головній гілці (main, master).
Процес збірки послідовно виконує:

Запуск лінтера Ruff (формат виводу адаптований для GitHub для інтеграції коментарів у Pull Requests).

Перевірку форматування коду (порушення форматування призводить до статусу failed).

Перевірку статичних типів за допомогою Mypy.

Виконання модульних та інтеграційних тестів за допомогою pytest (база даних імітується у пам'яті).
Злиття (merge) коду можливе лише у випадку зеленого статусу (Green Build) всього пайплайну.

5. Скрипт комплексної перевірки
Для локальної перевірки коду розроблено зручний механізм агрегації команд. Перед відправкою коду до віддаленого репозиторію рекомендується використовувати команду:

make check-all
Це автоматично запустить форматування, лінтинг, перевірку типів та повний набір тестів.
